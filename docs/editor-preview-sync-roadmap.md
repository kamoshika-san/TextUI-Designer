# エディタ・プレビュー双方向同期ロードマップ（MVP起点）

## 目的
- YAML編集中のカーソル位置とプレビュー選択の往復を、段階的に製品へ導入する。
- 既存の Jump-to-DSL 体験を壊さず、編集ループを短縮する。

## 現状境界（要点）
- WebView 側は `jump-to-dsl` / `preview-navigate` 系メッセージを送信できる。
- 拡張側は `webview-message-handler` でメッセージ受信し、エディタ選択へ反映できる。
- 逆方向（エディタカーソル変化をプレビュー選択へ反映）は標準経路が不足している。

## 段階計画
### Phase A（MVP: エディタ→プレビュー片方向）
- エディタ選択イベントを購読し、現在要素の DSL パスを WebView へ通知する。
- プレビュー側で該当コンポーネントを一時ハイライト表示する。
- 非対応パスは無視し、警告ログのみに留める。

### Phase B（プレビュー→エディタ整流）
- 既存 `jump-to-dsl` を統合し、フォーカス復帰とスクロール位置維持を標準化する。
- 連続クリック時のスロットリングを導入し、過剰ジャンプを防ぐ。

### Phase C（双方向安定化）
- 双方向イベントの循環抑止（originタグ・タイムスタンプ）を導入する。
- 大規模DSL時の性能計測（更新遅延・メッセージ頻度）を追加する。

## 子チケット候補（優先順位・依存）
1. **T-215-A**: エディタ選択イベントから DSL パス通知を追加（Phase A, 最優先）
2. **T-215-B**: プレビュー側ハイライト受信・表示コンポーネント追加（Phase A, A依存）
3. **T-215-C**: `jump-to-dsl` 連続操作時の整流とフォーカス維持（Phase B, A/B依存）
4. **T-215-D**: 双方向循環抑止ガード導入（Phase C, C依存）
5. **T-215-E**: 性能計測と閾値アラートの最小導入（Phase C, D依存）

## リスクと緩和策
- **パフォーマンス劣化**: 選択イベント連打でWebView更新が過密化する。  
  → 50-100msデバウンスと差分通知のみ送信。
- **DSL曖昧性**: 同型コンポーネントが多い画面で解決先が不安定。  
  → `dslPath` 優先、解決不可時は index 補助キーでフォールバック。
