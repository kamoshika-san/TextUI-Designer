## 2. ファイル切り替え・テーマまわりで試したが打ち切りにした変更

以下の変更は、別の不具合（Loading のまま、テーマが効かない、セレクタのちらつき）を招いたため**打ち切り**。HEAD に戻す。

### 2.1 ファイル切り替え時の WebView ちらつき対策（打ち切り）

- **webview-utils.ts**  
  `update` / `json` 転送時に `fileName` を `postMessage` に含めるように変更。
- **use-webview-messages.ts**  
  `applyDslUpdate(dsl, sourceFileName?)` の第2引数を追加し、受信時に `fileName` を渡す。
- **webview.tsx**  
  `lastSourceFileRef` を導入。`sourceFileName` が前回と異なる場合はマージせず `setJson(incomingDsl)` のみ実行し、`prevComponentsRef` を null にリセット。
- **webview-update-manager.ts**  
  - `setLastTuiFile` で **先に** `this.lastTuiFile = filePath` を代入してからキューに投入（タスク実行時に古いファイルの内容が送られないようにする）。  
  - ファイル変更時に `clearDebounceTimer()` を呼び、古いファイル用のデバウンス送信をキャンセル。  
  - ファイル変更による即時更新では `sendYamlToWebview(true)` に変更（自動プレビューオフでも送信するため）。
- **use-webview-messages.ts**  
  json/update を 50ms デバウンスし、連続メッセージは最後の1件だけ適用。初回のみデバウンスせず即時適用（Loading のままになる問題への対応）。

### 2.2 テーマ切り替えまわり（打ち切り）

- **use-webview-messages.ts**  
  `theme-variables` を 40ms デバウンスし、直近の1件だけ `applyThemeVariables` に渡すように変更。
- **webview-message-handler.ts**  
  テーマ切り替え時の `sendYamlToWebview(false)` を削除（DSL 再送による再描画でちらつくため）。

→ 結果として **テーマが適用されなくなる** および **テーマ選択セレクタがちらつく** などの副作用。テーマ周りの変更は取りやめ。

## 3. 得られた知見

1. **拡張側のキューと状態の順序**  
   キューに投入したタスクが `sendYamlToWebview` 内で `this.lastTuiFile` などを参照する場合、**キュー投入前に** `lastTuiFile` を更新しておかないと、タスクが同期的（またはすぐ）実行されたときに古いファイルの内容が送信される。ファイル切り替え時は「状態を先に更新 → その後にキュー投入」の順序が重要。

2. **自動プレビューオフ時のファイル切り替え**  
   ファイル変更による即時更新で「プレビューを必ず更新したい」場合は、`sendYamlToWebview(false)` だと「自動プレビューが無効なら送信しない」でスキップされる。明示的なタブ切り替え更新では `sendYamlToWebview(true)` のように強制送信が必要。

3. **デバウンスと初回メッセージ**  
   初回の json/update もデバウンスで遅延させると、React の Strict Mode などで effect が二重実行されたときに cleanup で `clearTimeout` が走り、初回メッセージが一度も適用されず「Loading のまま」になるリスクがある。**初回はデバウンスせず即時適用**する設計が安全。

4. **テーマと CSS 変数の適用**  
   テーマは `theme-variables` で CSS 変数を差し替えるだけで反映される。一方で、`theme-variables` のデバウンスやテーマ切り替え時の `sendYamlToWebview` 削除は、**テーマが効かなくなる**・**テーマ選択 UI のちらつき**といった別の副作用を招いた。テーマ・テーマ選択 UI の変更は影響範囲を小さくし、慎重に評価する必要がある。

5. **fileName による「ファイル切り替え」検知**  
   拡張から `fileName` 付きで update を送り、WebView 側で「前回と fileName が違うときはマージせず全置換」にすると、ファイル切り替え時の表示の混在は防げる。ただし、その前後のデバウンス・初回即時適用・拡張側の送信タイミングと組み合わせると、Loading やテーマまわりで別問題が出たため、一連の変更は打ち切りとした。

---

## 4. 今後の再挑戦時の方針（メモ）

- ファイル切り替えのちらつきだけを抑えたい場合は、**拡張側の送信順序・状態更新順序の修正**（lastTuiFile の先更新、ファイル変更時のデバウンスキャンセル、必要なら forceUpdate: true）に絞り、**WebView 側のデバウンスや fileName による全置換は入れない**、あるいは最小限にして試すとよい。
- テーマのちらつきは、**theme-variables のデバウンス**や **テーマ切り替え時の sendYamlToWebview 削除**は、テーマ適用やセレクタの表示に悪影響が出たため、別の方法（例: テーマ適用を requestAnimationFrame で1フレーム遅延させる、など）を検討する必要がある。
- 本チケットで残す計測・差分マージの最適化は、上記の打ち切り変更とは独立しているため、HEAD に戻したあとも必要に応じて再適用できる。
