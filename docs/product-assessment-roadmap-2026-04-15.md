# TextUI Designer プロダクト評価・PMF仮説・開発ロードマップ

更新日: 2026-04-15

## 1. エグゼクティブサマリー

- **総合評価**: TextUI Designer は、**「YAML/JSON DSL で UI を定義し、VS Code 上でプレビューし、そのまま複数形式へエクスポートする」**という中核価値に関しては、すでに **ベータ後期〜初期商用化手前**の完成度にある。
- **機能成熟度**: プレビュー、DSL 検証、テーマ、ナビゲーションフロー、CLI、MCP まで揃っており、単機能ツールではなく **一連の制作ワークフローを持つ製品**になっている。
- **PMF の現状**: コードベースから確認できる範囲では、**技術的な解像度は高い一方で、市場検証の証跡はまだ弱い**。したがって現時点の判定は **pre-PMF** だが、狙う顧客を絞れば PMF 候補になり得る。
- **最重要戦略**: 「汎用 UI ビルダー」を目指すより、**AI と相性の良い UI DSL ワークベンチ**として、内部業務アプリ・PoC・仕様駆動 UI 生成に集中する方が勝ち筋がある。

### 評価根拠

本評価は、主に `README.md`、`package.json`、`CHANGELOG.md`、`sample/README.md`、`docs/test-matrix.md`、`docs/ci-quality-gate.md`、`docs/mcp-integration.md`、`.github/workflows/ci.yml`、`src/core/textui-core-engine.ts`、`src/services/command-catalog.ts`、`src/services/webview/webview-message-handler.ts` をもとに整理した。

## 2. コードベースから確認できる事実

### 2.1 中核価値は明確

README は、`.tui.yml` を書くとライブプレビューでき、HTML / React / Svelte / Vue へ出力できると明示している。単なるレンダラではなく、**定義 -> プレビュー -> エクスポート** の流れを製品メッセージとして打ち出せている。さらに、テーマ、`$include`、スキーマ検証、Jump-to-DSL、Flow Preview、MCP 連携まで案内されているため、表面的な MVP はすでに超えている。

### 2.2 実装の広がりは十分にある

`package.json` では拡張本体に加えて `media/`、`schemas/`、CLI の `bin`、MCP サーバーを含めて配布する構成になっている。設定項目も、プレビュー、診断、パフォーマンス、テンプレート、MCP 自動構成まで存在し、**製品表面の厚み**は高い。

また、`TextUICoreEngine` は UI 生成、検証、フロー検証、スキーマ参照、差分比較まで担っており、VS Code 拡張専用のコードではなく **再利用可能なコアエンジン** として整理されている。これは CLI / MCP / 将来の外部統合に拡張しやすい設計であり、プロダクト資産として強い。

### 2.3 品質の作り込みも進んでいる

テストマトリクスでは unit / integration / simulated E2E / regression / real host smoke を分けて定義している。`package.json` でも `test:all:ci`、`react-ssot-check`、import graph guard、SSoT チェックなどが整備され、CI 品質ゲート文書もある。これは「動く機能がある」だけでなく、**継続的に壊れにくくする運用設計**が進んでいることを示す。

## 3. 完成度評価

### 3.1 完成度の総評

完成度は 1 つの数値ではなく、**技術完成度**と**市場完成度**で分けて見るべきである。

| 観点 | 評価 | 判断 |
|---|---:|---|
| コア価値の実装完成度 | 4.5 / 5 | DSL 記述、ライブプレビュー、コード出力という中核体験は成立している |
| 周辺機能の充実度 | 4.0 / 5 | Theme / Include / Flow / CLI / MCP まで揃い、用途の幅が広い |
| 品質・保守運用の成熟度 | 3.8 / 5 | テスト層、品質ゲート、SSoT ガードは強い |
| リリース運用の完成度 | 3.0 / 5 | 実ホスト検証と WebView ビルド運用に改善余地が残る |
| 市場検証の成熟度 | 2.0 / 5 | コードベースだけでは利用継続率・導入速度・刺さる顧客像の確証が弱い |

**結論**:

- **技術完成度**: 高い
- **製品完成度**: 中〜高
- **市場完成度**: まだこれから

### 3.2 機能別成熟度

| 機能領域 | 判定 | 根拠 |
|---|---|---|
| DSL オーサリング | Mature | スキーマ、サンプル、コンポーネント群、テンプレート/Include が揃っている |
| プレビュー | Mature | Live preview、Jump-to-DSL、更新表示、テーマ切替がある |
| エクスポート | Growth | 複数形式に対応する一方、README / 設定値の表現に不整合がある |
| ナビゲーションフロー | Growth | v2 ガイド、フローサンプル、プレビュー導線があるが、運用/UX の詰めが必要 |
| CLI / MCP | Strong differentiator | AI ワークフローに接続できる点は非常に強い |
| QA / CI | Mature | テスト層・品質ゲート・リリース解釈が文書化されている |
| PMF 検証基盤 | Immature | 市場計測や顧客学習の仕組みはコードベースからは十分に見えない |

## 4. 成熟度を押し上げている強み

### 4.1 プロダクトの価値提案が一貫している

「YAML/JSON DSL を中心に据え、同一ソースから Preview / Export / CLI / MCP をつなぐ」設計は、単なる UI 表示ツールではなく **UI 仕様の実行環境**としての位置づけを作れている。

### 4.2 AI 時代との相性が良い

MCP 同梱により、AI エージェントが `generate_ui`、`validate_ui`、`capture_preview` などを直接呼べる。これは「人が YAML を書く」だけでなく、**AI が画面仕様を生成・検証・可視化する基盤**として展開できることを意味する。ここは同種の DSL ツールの中でも差別化しやすい。

### 4.3 コードベースの再利用性が高い

コアロジックが `TextUICoreEngine` にまとまり、拡張・CLI・MCP のアダプタ構造になっているため、将来的に SaaS / CI / 生成パイプラインへ広げる余地が大きい。これはロードマップ上の選択肢を広げる。

## 5. いまのままでは PMF に届きにくい理由

### 5.1 「誰の何を最短で解決するのか」がまだ広い

現状は、UI 設計、プレビュー、エクスポート、フロー、Diff、MCP までカバーしているため、機能群としては強い。一方で、**最初に勝ち切るべき顧客像**が広すぎる。汎用 UI ツールとして広く売ろうとすると、Figma / Storybook / low-code / frontend scaffolding 全方位と競争しやすい。

### 5.2 ドキュメントと設定の不整合が信頼を削る

現状のコードベースには、以下のような整合性リスクがある。

- README ではエクスポート形式を `html / svelte / vue` と説明している一方、`package.json` の `textui-designer.export.defaultFormat` は `html / react / pug` を列挙している。
- README の MCP 接続例は `dist/mcp/server.js` を示している一方、実スクリプトは `out/mcp/server.js` を使っている。
- WebView メッセージハンドラは `textui-designer.openFlowPreview` を呼ぼうとしているが、コマンドカタログ側にはその定義が見当たらない。

これらは致命的な未完成ではないが、**「使ってみたら説明と違う」**を招きやすく、PMF 前に潰すべき信頼障害である。

### 5.3 出荷の最終安心感がまだ弱い

CI 文書上は real host smoke がデフォルトの緑判定に含まれておらず、Build Extension ジョブでも `build-webview` は実行されていない。つまり、**コード品質は高いが、出荷直前の UX とアセット整合を常に強制できているわけではない**。初期ユーザー獲得期は、この差が体験品質に直結する。

## 6. PMF の可能性評価

### 6.1 PMF 可能性の判定

**判定: 中程度以上の可能性あり。ただし条件付き。**

その条件とは、以下の 2 点である。

1. **狙う市場を狭くすること**
2. **「AI と一緒に UI を仕様駆動で組み立てる」体験を主戦場にすること**

### 6.2 もっとも刺さりやすい顧客セグメント

優先順位順に、次の順で狙うのがよい。

1. **内部業務アプリを素早く作りたい開発チーム**
   - 要件はあるが、Figma と実装を往復したくない
   - YAML / Git / VS Code に抵抗がない
   - 画面数が多く、フロー定義やテンプレート化の価値が高い

2. **AI エージェントと協働して PoC を量産したいチーム**
   - MCP で DSL 生成と検証を自動化したい
   - 「チャットで仕様を書く -> UI たたき台が出る」を重視する

3. **デザイン専任が薄いプロダクトチーム**
   - 実装前に構造化 UI を固めたい
   - 完全なピクセル精度より、速度と整合性を重視する

### 6.3 逆に相性が悪い市場

- 完全ノーコードを期待する非エンジニア中心市場
- ブランド表現や高度なアニメーションが重要なコンシューマー向け UI
- デザインシステム統合や高度なカスタムコンポーネントを最初から強く求める大企業基盤案件

これらに早期から広げると、コアの強みがぼやける。

## 7. 推奨プロダクト戦略

### 7.1 取るべきポジショニング

TextUI Designer は、**「VS Code 上で動く、AI ネイティブな UI DSL ワークベンチ」**として位置づけるべきである。

訴求メッセージは、次の 3 点に絞る。

1. **仕様を YAML で持てる**
2. **その場で画面確認できる**
3. **AI と一緒に生成・検証・出力できる**

### 7.2 戦略上の優先順位

優先順位は次の通り。

1. **Activation 強化**
   - 初回体験を短くする
   - README / サンプル / コマンド導線を揃える

2. **Trust 強化**
   - Preview と Export の一貫性
   - フロー遷移や Back-to-flow の信頼性
   - 配布アセットと CI の整合

3. **PMF 学習**
   - どのユースケースで継続利用されるかを検証
   - 「内部ツール」「PoC」「AI 生成 UI」のどれが最も刺さるか比較

4. **Expansion**
   - 顧客像が定まった後にのみ、コンポーネント拡張や SaaS 化を検討

### 7.3 いま優先すべきでないもの

- コンポーネント種類の無制限拡張
- フレームワーク対応の横展開だけを増やすこと
- 汎用 low-code 市場への早期進出
- 高度なコラボレーション機能の先行実装

今は「広げる」より **刺さる導線を尖らせる** 方が重要である。

## 8. フェーズ別ロードマップ

以下はカレンダーではなく、**到達基準ベース**のロードマップである。

### Phase 1: プロダクトの真実を揃える

目的: **説明・設定・挙動の不整合をなくし、初回体験を安定させる**

必須項目:

- README、設定、コマンド、CHANGELOG の整合を取る
- MCP 起動パス表記を `out/` 基準へ統一する
- export format の製品メッセージを整理する
- `openFlowPreview` 周辺の導線を実装または文書と合わせる
- 3 本のゴールデンパスを定義する
  - 初回 `.tui.yml` プレビュー
  - Flow preview からページ遷移
  - MCP で scaffold -> validate -> capture

完了条件:

- 新規ユーザーが README どおりに試して迷わない
- 「書いてあることと動くことが違う」が主要導線で消えている

### Phase 2: 使ってよい安心感を作る

目的: **ベータ品質から、チーム導入可能な品質へ上げる**

必須項目:

- real VS Code smoke をリリース判定に近づける
- Build Extension 系 CI に `build-webview` を組み込むか、運用上の正本を明確化する
- preview / export / capture の整合性を、代表サンプルで見える化する
- サンプルベースの受け入れチェックを、ユーザーシナリオ単位で整理する
- フロー機能の UX を安定化する

完了条件:

- リリース候補の安心感を、開発者だけでなく利用者視点でも説明できる
- 代表ユースケースで「壊れにくい」が定常化する

### Phase 3: PMF 仮説を検証する

目的: **どの顧客セグメントに最も刺さるかを、実利用で学習する**

必須項目:

- ターゲットを 2 セグメントまでに絞る
  - 内部業務アプリ
  - AI 併用 PoC / プロトタイピング
- ユースケース別のテンプレート・サンプル・プロンプトを用意する
- 「初回成功までの時間」「継続利用」「生成後の手修正量」を観測する
- インタビューまたは導入伴走で、詰まる地点を収集する

完了条件:

- どのユーザーに最も刺さるかが言語化できる
- 機能追加より onboarding / trust / AI 導線の方が効くのかを判断できる

### Phase 4: 拡張と事業化判断

目的: **勝ち筋が見えた後にのみ、投資先を広げる**

候補:

- design system 連携
- カスタムコンポーネント拡張 API
- SaaS / Web 管理画面
- チーム共有・レビュー導線
- 商用パッケージ化、導入支援、テンプレート販売

着手条件:

- Phase 3 で明確な主要顧客像が定まっている
- 継続利用の理由が「機能数」ではなく「ワークフロー価値」で説明できる

## 9. 直近で追うべき KPI

PMF を検証するには、コード品質指標だけでは不十分である。今後は次の KPI を持つべきである。

### 9.1 Activation KPI

- 初回プレビュー成功率
- 初回エクスポート成功率
- 初回 MCP 利用成功率
- サンプルから自作 DSL へ移行できた割合

### 9.2 Value KPI

- 1 週間以内の再利用回数
- 1 プロジェクトあたりの `.tui.yml` / `.tui.flow.yml` 継続作成数
- capture / export / scaffold 利用比率
- 生成後にどれだけ手修正が必要だったか

### 9.3 Trust KPI

- Preview と Export の差異報告件数
- Flow 遷移まわりの不具合率
- README 記載と実装差異に起因する問い合わせ件数

## 10. 最終提言

TextUI Designer は、**技術的には「もう使える」段階にあるが、市場的には「どこに最も刺さるかを今から決める」段階**である。

したがって今後の意思決定は、次の原則で行うべきである。

1. **市場を広げる前に、対象顧客を絞る**
2. **新機能追加より、初回成功体験と信頼性の整合を優先する**
3. **AI/MCP を主戦場に据え、仕様駆動 UI 生成の体験を尖らせる**
4. **PMF が見えるまでは、機能の幅ではなく導線の深さで勝つ**

最も有望な勝ち筋は、**「内部ツール / PoC 向けの、AI ネイティブ UI DSL ワークベンチ」**として磨き込むことである。
