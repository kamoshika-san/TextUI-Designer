# 生成AI活用 画面設計ワークフロー ロードマップ（v1）

最終更新: 2026-03-08

## 1. 目的

TextUI Designer の「生成AIで画面設計→検証→出力→運用」フローを、PoC品質から運用品質へ引き上げる。

## 2. 現状サマリ

- 強み
  - MCP サーバー経由で `generate_ui / validate_ui / explain_error / run_cli` を利用可能
  - CLI の `validate / plan / apply / export / import(openapi)` が整備済み
  - unit/integration/e2e/regression の包括テストが存在
- 課題
  - MCP のヘッダーレス連続メッセージ処理の堅牢性をさらに強化したい
  - State 運用は MVP レベルで、ロック/リモートバックエンドは次段階
  - リリース前チェック（生成アセット差分や配布物整合）の運用ルールを明文化したい

## 3. 到達目標（12週間）

### プロダクト目標

1. 生成AIワークフローの成功率を高める（再実行不要率を向上）
2. CI/CD での差分検知・適用を安定運用できる
3. 「使い始めるまでの時間」を短縮する（初期セットアップ摩擦を最小化）

### KPI（目安）

- AIフロー成功率（初回で validate 通過）: **70% → 90%**
- `plan` 実行のノイズ差分率（不要差分）: **20% → 5%以下**
- MCP起因エラー率（セッションあたり）: **3%未満**
- 初回導入時間（README手順のみ）: **30分 → 10分**

## 4. フェーズ計画

## Phase 1（1〜2週）: MCP安定化と可観測性

### 優先度: 最優先

### 施策

1. MCPトランスポート改善
   - ヘッダーレス入力を 1 メッセージ単位で安全に分割処理
   - 複数JSON連結時のパーサ戦略を追加（改行区切り/逐次デコード）
2. MCP監視ログの標準化
   - requestId / method / latency / error code を共通形式で記録
3. スモークテスト追加
   - `initialize → tools/list → tools/call(generate_ui)` の最短導線をCI化

### 成果物

- MCP耐障害性の改善PR
- MCPスモークテスト（自動実行）
- トラブルシューティング更新

### 完了条件（DoD）

- 連続ヘッダーレス入力ケースのテスト追加・全件パス
- MCP関連既存テストの回帰なし

## Phase 2（3〜6週）: AI設計フロー品質向上

### 優先度: 高

### 施策

1. 生成品質改善
   - `generate_ui` のデフォルト補完ルールをユースケース別に拡張
   - `explain_error` の修正提案テンプレートを改善（必須項目不足、型違反、token関連）
2. OpenAPI import 強化
   - oneOf/allOf/array/object 混在の実案件寄りケースを拡充
3. サンプル拡充
   - AI入力例（要件文）→ DSL → 出力コードの成功パターンを追加

### 成果物

- 生成品質改善PR（Core Engine）
- OpenAPI importer の追加テスト
- 「AIプロンプト例集」ドキュメント

### 完了条件（DoD）

- 追加ケースで `validate` 通過率が改善
- 回帰テスト全パス

## Phase 3（7〜9週）: 運用機能（State/CI）強化

### 優先度: 中〜高

### 施策

1. Stateロック戦略
   - 楽観ロック強化（競合再試行・エラーメッセージ改善）
2. バックエンド方針決定
   - ローカルstate継続 or リモートstate（S3/GCS等）を設計判断
3. CIテンプレート強化
   - `validate`/`plan`/`apply` の標準ジョブ化（差分コメント運用含む）

### 成果物

- State競合制御改善PR
- 運用設計メモ（ローカル/リモート比較）
- CIテンプレート改訂版

### 完了条件（DoD）

- 競合時の失敗理由が再現可能・追跡可能
- CIで差分検知運用が定常化

## Phase 4（10〜12週）: DX最適化とリリース品質

### 優先度: 中

### 施策

1. 初期導入導線の最適化
   - README「最短5分導入」を整備
2. リリースゲート整備
   - 配布物チェックリスト（ビルド成果物、MCP起動、CLI基本コマンド）
3. 失敗時の復旧体験改善
   - 代表障害パターンと解決フローをガイド化

### 成果物

- 導入ガイド改訂
- リリースチェックリスト
- 障害時ランブック

### 完了条件（DoD）

- 新規メンバーが手順のみで導入完了
- リリース前手順の抜け漏れ防止が機能

## 5. 優先バックログ（横断）

1. MCP連続メッセージ処理の堅牢化
2. `generate_ui` のドメイン別プリセット（業務管理/EC/管理画面）
3. `run_cli` の実行結果要約（失敗理由の構造化）
4. `plan` 差分の可読性向上（属性レベル表示の拡張）
5. サンプル案件（小/中/大規模）セットの整備

## 6. リスクと対策

- リスク: 生成品質改善で既存DSL互換を壊す  
  対策: 既存サンプル全件のスナップショット検証を必須化

- リスク: MCP互換性改善でクライアント差異に再発  
  対策: Cursor想定・標準JSON-RPC想定の両方で自動テスト

- リスク: State機能強化で複雑化  
  対策: v1仕様（最小責務）を維持し、拡張はオプトインで段階導入

## 7. 直近2週間の着手順（推奨）

1. MCP連続入力の再現テスト追加
2. トランスポート修正（段階的）
3. `generate_ui` と `explain_error` の改善を小分けで適用
4. 追加ケースで回帰確認（unit + e2e + regression）
5. ドキュメント更新・運用反映
