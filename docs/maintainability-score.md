**現状スコア: 82 / 100（B+）**

- 基盤品質（TypeScript strict / lint / 多層テスト）は強い
- ただし、**大きなファイルへの責務集中**と**品質ルールの段階的強化不足**が将来コストの主因

## 強み（維持すべき点）

1. **品質ゲートの入口が揃っている**
   - `npm test` 実行前に `compile + lint` が必ず走る。
2. **テスト実行の再現性が高い**
   - `tests/setup.js` で `vscode` をモック化し、環境依存を減らしている。
3. **機能提供面の成熟度が高い**
   - extension / CLI / MCP / exporter と複数チャネルを既に統合。

---

## 主な負債（優先度順）

1. **巨大ファイル依存（最優先）**
   - 300〜460行級ファイルが複数あり、変更時の影響分析とレビューが重くなりやすい。
2. **TSコンパイラチェックの未活用**
   - `noImplicitReturns` など、バグ予防に効くフラグが未有効。
3. **テスト命名・失敗時切り分けの標準化不足**
   - スイートはあるが、運用ルール（失敗分類/命名規約）の明文化余地がある。

---

## 改善提案（実行しやすい順）

### フェーズ1（1〜2週間）: 低リスクで効く施策

- `tsconfig` で次を段階導入（まずCI warning運用→fail化）
  - `noImplicitReturns`
  - `noFallthroughCasesInSwitch`
  - `noUnusedParameters`
- テスト命名規約を追加
  - 例: `feature__condition__expected` 形式
- PRテンプレートに「変更影響範囲」「ロールバック方法」を追記

**期待効果:** 不具合混入率の低下、レビュー時間の短縮。

### フェーズ2（3〜5週間）: 構造改善

- 350行超ファイルを上位3つから分割
  - ルール: 「I/O層」「ドメイン処理」「表示/整形」を分離
- exporter系に共通インターフェースとユーティリティ整理を追加

**期待効果:** 変更容易性・テスト容易性の上昇（+4〜6点想定）。

### フェーズ3（6〜8週間）: 運用定着

- CIのブランチ保護に `test:all:ci` を必須化
- 回帰テストの失敗分類タグ（schema / exporter / preview / mcp）を導入
- 月次で「変更失敗率」「平均レビュー時間」を可視化

**期待効果:** 品質のばらつきを抑え、保守コストを継続的に低減。
