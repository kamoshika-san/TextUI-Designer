# 品質ゲートと「緑の main」（運用ポリシー）

本ページは、アーキテクトロードマップ **フェーズ 0** に沿い、**作業ツリーで品質コマンドが赤のままではマージ判断を保留する**という前提を、日々の開発フローで揃えるためのメモです。CI の必須チェック名や branch protection の手順は別ドキュメントに分離しています。

## ローカルで最低限どのコマンドを緑にしてから push / PR するか

- **既定（`main` 向け PR を出す前）**: `npm test` を **成功**させる。
  - `package.json` 上、`npm test` の前に `pretest` が走り、`compile` → `typecheck:strict` → `lint` のあと **ユニットテスト**が実行されます。これがローカルでの **最小の一括ゲート**です。
- **変更の影響が広い場合（推奨）**: `npm run test:all` または CI と同型の `npm run test:all:ci` を追加で実行し、統合・E2E・回帰まで含めて確認する。
- **マージ判断**: `main` へのマージは、リポジトリで定めた **必須 CI チェック**が緑であることを前提とします（詳細は下記「関連ドキュメント」）。

## WIP（作業中）の扱い・コミットの粒度

- **論理単位でコミット**する（1 PR に無関係な変更を混ぜない）。
- **ローカルで `npm test` が赤の状態**を「そのまま push して後で直す」前提にしない。どうしても共有が必要な場合は **ドラフト PR** またはチームで合意したラベル・説明で意図を明示する。
- ロードマップの趣旨: *作業ツリーで `npm test` / `npm run test:all` が赤のままではマージ判断を保留* — チーム全員が同じ基準で止まれるようにする。
